# Как я перестал каждый раз объяснять Codex проект заново: простая папка `docs/ai_context`

**TL;DR.** У Codex и Claude в новом чате часто одна и та же проблема: они не знают проект. Где главный код, где опасные файлы, что уже решили, чем проверять результат. Я перестал искать секретный промпт и сделал обычную техкарту внутри репозитория: `docs/ai_context/`. Там шесть коротких файлов: карта проекта, текущая цель, опасные зоны, проверка, журнал решений и итог сессии. Весь набор выложил в [github.com/DevKitRU/ai-context-kit](https://github.com/DevKitRU/ai-context-kit), MIT.

Под катом: зачем это нужно, как устроены файлы, что туда писать, что туда нельзя писать, как я проверял на своих проектах и почему это полезнее, чем огромный промпт на 200 строк.

---

## Контекст

Я не профессиональный разработчик. Основная работа у меня из другой области, но за последние месяцы я начал много работать с Codex и Claude Code: сайты, телеграм-боты, маленькие автоматизации, большой рабочий репозиторий с API, PWA, админкой и продакшн-деплоем.

Проблема проявилась быстро.

Новая вкладка агента каждый раз начинает почти с нуля.

Я открываю проект и пишу:

> поправь форму заявки, не трогай авторизацию

А агенту сначала надо понять:

- где эта форма;
- где API;
- где тесты;
- где `.env`;
- что уже делали вчера;
- какие части проекта лучше не трогать;
- чем проверить результат.

Если проект маленький, это терпимо.

Если проект большой, агент начинает тратить контекст на разведку. Иногда полезную, иногда лишнюю. Иногда он читает не те файлы. Иногда просит показать `.env`. Иногда предлагает деплой там, где нужна только правка текста.

Это не потому что модель плохая. Это потому что мы кидаем ее на объект без карты.

## Что я пробовал

### Один большой промпт

Пробовал держать в начале чата большой промпт:

```text
Вот проект. Тут бот. Тут сервер. Тут база. Тут тесты. Не трогай это. Проверяй вот так.
```

Работает до первого изменения.

Потом промпт устаревает. Ты переносишь файл, меняешь команду тестов, добавляешь новый сервис, а старый промпт продолжает жить в заметках. Через неделю он уже частично врет.

### Только README

README нужен людям. Но README редко отвечает на вопросы агента:

- какие файлы опасные;
- какие тесты запускать именно для этой зоны;
- что сейчас не является целью;
- какие решения уже приняты;
- что было сделано в прошлой сессии.

README обычно объясняет проект внешнему читателю. Агенту нужна рабочая карта.

### Память агента

Память помогает, но у нее другой масштаб.

Память хранит опыт между сессиями: мои предпочтения, проекты, общие правила. Но конкретный проект должен сам носить свою карту рядом с кодом.

Иначе получается странно: код поменяли, а память где-то отдельно. Новая вкладка начинает спорить с реальностью.

## Что получилось

Я сделал маленький слой внутри репозитория:

```text
docs/ai_context/
├── PROJECT_MAP.md
├── CURRENT_GOAL.md
├── DANGER_ZONES.md
├── VERIFICATION.md
├── DECISIONS.jsonl
└── SESSION_SUMMARY.md
```

Плюс короткая строка в `AGENTS.md`:

```md
For faster cold start, read `docs/ai_context/PROJECT_MAP.md` next, then only the relevant `docs/ai_context/*` file for the task.
```

Если проект использует Claude Code, такую же строку можно добавить в `CLAUDE.md`.

Смысл простой: агент не читает весь репозиторий сразу. Он сначала читает карту, потом нужную зону, потом конкретные файлы.

Это называется красивыми словами `context engineering`, но по сути это обычная инженерная гигиена.

Как лист на объекте:

- где щиток;
- где вода;
- где газ;
- что не трогать без разрешения;
- чем проверить, что после работы ничего не течет.

## Файл 1. `PROJECT_MAP.md`

Это главный вход.

В нем должно быть коротко:

- что это за проект;
- какие папки главные;
- где точка входа;
- где тесты;
- куда идти для типовых задач;
- какие команды поиска помогают.

Пример:

```md
# PROJECT_MAP

## Что это

Telegram bot для обработки заявок.
Основной код в `src/`.
SQLite база рядом, но агенту не нужно читать ее напрямую.

## Быстрый вход

1. Прочитай этот файл.
2. Если задача про секреты, продакшн или данные пользователей, открой `DANGER_ZONES.md`.
3. Для проверки смотри `VERIFICATION.md`.

## Карта файлов

| Путь | Роль | Когда читать |
| --- | --- | --- |
| `src/bot.py` | Telegram handlers | меняем поведение бота |
| `src/services/` | бизнес-логика | меняем обработку заявки |
| `tests/` | тесты | проверяем правку |
| `.env` | секреты | не читать и не печатать |
```

Это не документация на 40 страниц.

Хороший `PROJECT_MAP.md` может быть на 50 строк. Главное, чтобы новая вкладка не начинала с хаотичного `ls -la` и чтения всего подряд.

## Файл 2. `CURRENT_GOAL.md`

У агента часто есть соблазн сделать "заодно".

Попросили поправить форму, он полез чинить архитектуру.

Попросили добавить тест, он предложил миграцию базы.

`CURRENT_GOAL.md` помогает держать рамку.

Пример:

```md
# CURRENT_GOAL

Дата: 2026-05-13.

## Текущая цель

Стабилизировать обработку заявок.

## Не цель сейчас

- Не менять авторизацию.
- Не трогать deploy.
- Не читать `.env`.
- Не делать commit без команды.

## Следующий полезный шаг

Добавить тест на повторную отправку формы.
```

Этот файл особенно полезен, когда проект открыт в нескольких вкладках. Одна вкладка чинит UI, другая смотрит тесты, третья готовит деплой. Без текущей цели они быстро начинают мешать друг другу.

## Файл 3. `DANGER_ZONES.md`

Это самый важный файл для безопасности.

Там не должно быть секретов. Там должны быть правила вокруг секретов.

Пример:

```md
# DANGER_ZONES

## Секреты

- Не читать `.env`.
- Не печатать tokens, passwords, API keys, private keys.
- Не коммитить DB, logs, sessions, uploads.

## Production

- Не деплоить без отдельной команды.
- Не менять nginx, Docker, ports, cron, systemd как побочный эффект.

## Данные пользователей

- Не печатать личные сообщения, телефоны, email, user IDs без причины.
- Если нужна структура, использовать schema или fake data.
```

Важный момент: агенту не надо знать значение токена, чтобы знать, что токен трогать нельзя.

Плохо:

```text
BOT_TOKEN=123456789:...
```

Нормально:

```text
BOT_TOKEN лежит в `.env`. Не читать и не печатать `.env`.
```

## Файл 4. `VERIFICATION.md`

Частая проблема с агентами: правка сделана, но проверка не та.

Например, в проекте есть Python, Node и Docker. Агент запускает `npm test`, хотя менял Python API. Или запускает весь набор тестов на 20 минут, хотя достаточно одного файла.

`VERIFICATION.md` дает карту проверок.

Пример:

````md
# VERIFICATION

## Базово

```bash
git status --short
git diff --check
```

## Python

```bash
python3 -m py_compile src/bot.py
python3 -m pytest
```

## Docs-only

```bash
git diff --check
```

## Safety checklist

- `.env` не читался и не печатался.
- Секреты не попали в diff.
- Runtime files не попали в git.
- Проверки соответствуют зоне правки.
````

Этот файл экономит время и снижает риск "вроде работает".

## Файл 5. `DECISIONS.jsonl`

Это журнал решений.

Почему JSONL, а не markdown?

Потому что агенту легко дописать одну строку. И человеку легко читать diff.

Пример:

```jsonl
{"date":"2026-05-13","decision":"Use docs/ai_context as the project context layer.","reason":"New AI-agent sessions need a small map, danger zones, verification steps, and session continuity.","status":"accepted"}
{"date":"2026-05-13","decision":"Do not store secrets in context files.","reason":"Context must explain the project without exposing access.","status":"accepted"}
```

Это не замена ADR. Для больших архитектурных решений ADR лучше.

Но для маленьких "мы договорились делать так" JSONL удобен.

## Файл 6. `SESSION_SUMMARY.md`

Это сжатие прошлой сессии.

В конце работы агент пишет:

- что сделано;
- что выяснено;
- какие файлы менялись;
- какие проверки прошли;
- что осталось.

Пример:

```md
# SESSION_SUMMARY

Дата: 2026-05-13.

## Что сделано

- Добавлен persistent rate limit.
- Обновлены тесты.

## Что выяснено

- Старые JWT должны жить до истечения срока.
- Новая таблица сессий не должна ломать старых пользователей.

## Проверка

- `python3 -m pytest tests/test_auth.py` - ok.
- `git diff --check` - ok.

## Не сделано

- Deploy не делался.
```

Это защищает от деградации контекста.

Через день новый чат не должен заново раскопать все то же самое. Он читает summary и продолжает.

## Как это выглядит в работе

Нормальный старт задачи теперь такой:

```text
Прочитай AGENTS.md и docs/ai_context/PROJECT_MAP.md.
Не читай .env, DB, logs, uploads и приватные данные.
Найди файлы, которые относятся к форме заявки.
Скажи, какие проверки нужны, и потом внеси правку.
```

Агент открывает карту.

Видит, что форма в `webapp/`, API в `api/`, тесты в `tests/test_orders.py`, а `.env` не трогать.

Если задача про продакшн, читает `DANGER_ZONES.md`.

Если задача про проверку, читает `VERIFICATION.md`.

Если меняет решение, дописывает строку в `DECISIONS.jsonl`.

Если заканчивает сессию, обновляет `SESSION_SUMMARY.md`.

## Почему это работает

Тут три простых принципа.

### 1. Progressive disclosure

Не грузить все сразу.

Сначала карта. Потом нужная зона. Потом конкретные файлы.

Так агент меньше тратит контекст и меньше цепляет случайные старые куски.

### 2. Filesystem context

Контекст живет в репозитории.

Не в голове пользователя.
Не в старом чате.
Не в огромном промпте из заметок.

Код поменялся, рядом можно поменять карту.

### 3. Context compression

Итог сессии и решения сжимают историю.

Агенту не нужно перечитывать весь чат на 200 сообщений. Он видит короткий факт:

```text
Старые JWT оставили совместимыми.
Deploy не делался.
Следующий шаг: тесты миграции.
```

## Быстрый старт

Я сделал маленький open-source набор:

[github.com/DevKitRU/ai-context-kit](https://github.com/DevKitRU/ai-context-kit)

Установка в проект:

```bash
git clone https://github.com/DevKitRU/ai-context-kit.git
cd ai-context-kit
./scripts/init-ai-context.sh /path/to/project
```

Скрипт:

- создает `docs/ai_context/`;
- копирует шаблоны;
- создает `AGENTS.md`, если его нет;
- добавляет ссылку на `PROJECT_MAP.md`, если `AGENTS.md` уже есть.

Он не читает `.env`, базы, логи и приватные данные.

Можно и руками:

```bash
mkdir -p docs/ai_context
cp -R ai-context-kit/templates/docs/ai_context/* docs/ai_context/
```

## Что нельзя делать

Нельзя превращать `docs/ai_context` в свалку.

Плохие признаки:

- туда скопировали весь README;
- туда положили токены;
- туда вставили 500 строк чата;
- туда написали "всегда запускай все тесты";
- файл не обновляют после крупных изменений.

Хорошее правило:

```text
Каждый файл должен помогать новой вкладке быстрее и безопаснее сделать работу.
Если не помогает, удалить или сократить.
```

## Что показала обкатка

Я проверил формат на нескольких типах проектов:

- маленький Telegram bot;
- digest bot;
- проект с RAG и production secrets;
- большой web/SaaS repo с API, PWA, tenant boundary и deploy scripts.

Самый большой выигрыш оказался не в скорости.

Самый большой выигрыш в спокойствии.

Агент перестает каждый раз гадать:

- где вход;
- что опасно;
- что уже решили;
- что запускать перед финалом.

И я меньше ловлю моменты "стой, зачем ты полез в deploy".

## Ограничения

Это не магия.

`docs/ai_context` не исправит плохую архитектуру.

Не заменит тесты.

Не спасет от устаревшей информации, если ее не обновлять.

Не отменяет нормальный code review.

И точно не дает права класть секреты в репозиторий.

Но как базовая гигиена для работы с агентами оно работает.

## Связка с memory sync

Есть похожая, но другая задача: память между машинами.

Если у тебя Claude Code на Mac, Windows и VPS, можно синхронизировать личную память через git. Это тема отдельной статьи и отдельного репозитория.

`ai-context-kit` решает другой слой:

- memory sync хранит общую память пользователя;
- `docs/ai_context` хранит карту конкретного проекта.

Оба слоя не должны хранить секреты.

## Итого

Мне нравится думать об этом так:

AI-агенту нужен не "идеальный промпт".

Ему нужна нормальная рабочая среда.

Минимум:

- карта проекта;
- опасные зоны;
- проверка;
- журнал решений;
- итог сессии.

Это скучно. Зато работает.

Репозиторий:

[github.com/DevKitRU/ai-context-kit](https://github.com/DevKitRU/ai-context-kit)

Если у вас есть свой формат для агентного контекста в репозитории, интересно посмотреть. Особенно если он пережил не один маленький проект, а большой repo с продакшном, секретами и несколькими людьми.
