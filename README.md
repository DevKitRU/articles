# DevKitRU Articles

Здесь лежат статьи DevKitRU для Хабра и других площадок.

Это не витрина с пресс-релизами. Скорее рабочая тетрадь: черновики, мета для публикации, ссылки на репозитории и статус каждого текста.

Главный принцип простой:

> писать от реального опыта, показывать ошибки и давать человеку понятный следующий шаг.

## Как устроен репозиторий

Каждая статья лежит в своей папке:

```text
articles/
├── 01-my-vpn-kit/
│   ├── habr.md
│   ├── habr-meta.md
│   └── habr.html
├── 02-claude-memory-sync/
│   └── habr.md
└── 03-ai-context-kit/
    ├── habr.md
    └── habr-meta.md
```

`habr.md` - текст статьи.

`habr-meta.md` - служебные материалы для публикации: хабы, теги, короткое описание, Telegram-анонс, чеклист перед публикацией.

`images/` появится там, где нужны скриншоты или диаграммы.

## Как здесь пишутся тексты

Нормальный тон:

- личный опыт вместо рекламной подачи;
- конкретные симптомы вместо общих обещаний;
- ошибки и обходные пути, а не "всё получилось с первого раза";
- команды, цифры и проверки там, где они помогают;
- ссылка на репозиторий после объяснения пользы, а не вместо объяснения.

Плохой тон:

- "революционный фреймворк";
- "уникальная методология";
- "просто запустите и всё само заработает";
- скрывать грабли, потому что они портят картинку.

## Безопасность

В этот репозиторий нельзя класть:

- реальные `.env`;
- токены Telegram;
- API keys;
- OAuth secrets;
- приватные ключи;
- VPN configs;
- реальные production IP;
- логи с пользователями;
- базы данных;
- скриншоты с доступами.

Если в статье нужен пример секрета, он должен быть явно ненастоящим:

```text
ANTHROPIC_API_KEY=sk-ant-...
```

И рядом должно быть понятно, что это пример, а не рабочий ключ.
